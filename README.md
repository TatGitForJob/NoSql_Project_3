## 1. Краткое описание схемы базы данных

В проекте используется база данных `university` и одна основная коллекция `students`.

Структура документа:

```json
{
  "student_id": "S000001",
  "full_name": "Student 1",
  "group": "A-101",
  "faculty": "Engineering",
  "admission_year": 2024
}
```

Описание полей:

- `student_id` — уникальный идентификатор студента
- `full_name` — ФИО студента
- `group` — учебная группа
- `faculty` — факультет
- `admission_year` — год поступления

## 2. Репозиторий и состав проекта

## 4. Реализация шардинга

Для проекта была выбрана `MongoDB`, потому что в ней уже есть встроенная поддержка шардинга.

### Архитектура кластера

В проекте используется следующая конфигурация:

- `configsvr` — сервер конфигурации, хранящий метаданные кластера
- `shard1` — первый шард
- `shard2` — второй шард
- `mongos` — маршрутизатор запросов

Все сервисы запускаются через `docker-compose`, поэтому стенд легко повторить на другой машине.

### Выбор shard key

Коллекция `university.students` шардируется по полю `student_id` с использованием стратегии `hashed`.

Почему был выбран именно этот ключ:

- `student_id` есть у каждого документа;
- хэшированный ключ распределяет записи по shard-ам заметно ровнее;
- при последовательной генерации идентификаторов нет сильного перекоса в сторону одного узла;
- для учебного проекта такой вариант проще всего показать и объяснить.

### Что было реализовано

В init-скрипте кластера выполняются следующие действия:

1. Инициализация `replica set` для `configsvr`, `shard1` и `shard2`.
2. Добавление shard-ов в кластер через `mongos`.
3. Включение шардинга для базы `university`.
4. Создание индекса по `student_id`.
5. Шардирование коллекции `students`.

В результате получился рабочий кластер MongoDB, в котором данные реально раскладываются по двум shard-ам.

## 5. Реализация интерфейса на Python

Для работы с базой данных сделан минимальный консольный интерфейс на Python с использованием `pymongo`.

Подключение идёт через `mongos`, а не напрямую к shard-нодам.

Реализованные команды:

- `seed` — загрузка тестовых данных
- `add` — добавление одного студента
- `get` — получение одного студента по `student_id`
- `list` — вывод списка студентов с ограничением по количеству

Примеры использования:

```bash
python3 cli.py seed --count 1000
python3 cli.py add --student-id S999999 --full-name "Test Student" --group A-101 --faculty DE --admission-year 2026
python3 cli.py get --student-id S000000
python3 cli.py list --limit 3
```

Такого набора команд достаточно, чтобы показать базовую работу приложения с шардированной БД.

## 6. Нагрузочное тестирование

Для нагрузочного тестирования используется скрипт `benchmark.py`.

### Сценарий теста

Скрипт выполняет два основных типа операций:

- массовая загрузка `10000` документов в коллекцию `students`;
- выполнение `50000` точечных чтений по полю `student_id`.

Для сравнения использовались два окружения:

- одиночный MongoDB без шардинга;
- MongoDB-кластер с двумя shard-ами и `mongos`.

В обоих случаях запускался один и тот же сценарий: сначала загружался большой объём данных, потом выполнялось массовое чтение по `student_id`.

### Визуализация результатов нагрузочного тестирования

Таблица результатов:

| Режим | Загрузка 10000 документов | Чтение 50000 документов |
|---|---:|---:|
| Без шардинга | 0.131 c | 26.945 c |
| С шардингом | 27.5 c | 52.447 c |

Распределение данных по shard-ам:

| Shard | Документов | Доля |
|---|---:|---:|
| `shard1rs` | 5023 | 50.23% |
| `shard2rs` | 4977 | 49.77% |

## 7. Итог

По результатам теста на моём стенде одиночный MongoDB без шардинга оказался быстрее и на загрузке, и на чтении. Для такого объёма это нормально: у шардированного кластера есть дополнительные накладные расходы на `mongos`, метаданные и координацию нескольких узлов.

При этом сам шардинг отработал корректно. Данные распределились между двумя shard-ами почти поровну, без заметного перекоса. Это показывает, что ключ `hashed student_id` в данном случае выбран удачно.

Основной вывод такой: на относительно небольшом объёме данных шардинг не даёт выигрыша по времени, а может даже замедлять работу. Его смысл здесь не в ускорении маленького теста, а в подготовке системы к росту объёма данных и распределённому хранению. Чтобы увидеть выигрыш именно по времени, нужен стенд с заметно большим объёмом данных и более серьёзной нагрузкой.
