## 1. Краткое описание схемы базы данных

В проекте используется база данных `university` и одна основная коллекция `students`.

Структура документа:

```json
{
  "student_id": "S000001",
  "full_name": "Student 1",
  "group": "A-101",
  "faculty": "Engineering",
  "admission_year": 2024
}
```

Описание полей:

- `student_id` — уникальный идентификатор студента
- `full_name` — ФИО студента
- `group` — учебная группа
- `faculty` — факультет
- `admission_year` — год поступления

## 4. Реализация шардинга

В качестве СУБД выбрана `MongoDB`, так как она поддерживает встроенный механизм шардинга документов и позволяет сравнительно просто развернуть распределённую систему в учебном проекте.

### Архитектура кластера

В проекте используется следующая конфигурация:

- `configsvr` — сервер конфигурации, хранящий метаданные кластера
- `shard1` — первый шард
- `shard2` — второй шард
- `mongos` — маршрутизатор запросов

Все сервисы разворачиваются через `docker-compose`, что упрощает запуск и проверку проекта.

### Выбор shard key

Коллекция `university.students` шардируется по полю `student_id` с использованием стратегии `hashed`.

Причины выбора именно такого ключа:

- `student_id` есть у каждого документа и хорошо подходит в качестве устойчивого идентификатора;
- хэшированный ключ обеспечивает более равномерное распределение документов между shard-ами;
- последовательная генерация идентификаторов не создаёт явного перекоса в сторону одного shard-а;
- выбранный подход легко объяснить в рамках учебного проекта.

### Что было реализовано

В init-скрипте кластера выполняются следующие действия:

1. Инициализация `replica set` для `configsvr`, `shard1` и `shard2`.
2. Добавление shard-ов в кластер через `mongos`.
3. Включение шардинга для базы `university`.
4. Создание индекса по `student_id`.
5. Шардирование коллекции `students`.

В результате получен рабочий кластер MongoDB с горизонтальным распределением данных.

## 5. Реализация интерфейса на Python

Для работы с базой данных реализован минимальный консольный интерфейс на Python с использованием библиотеки `pymongo`.

Подключение выполняется к `mongos`, то есть клиент работает с кластером так, как должен работать реальный внешний потребитель.

Реализованные команды:

- `seed` — загрузка тестовых данных
- `add` — добавление одного студента
- `get` — получение одного студента по `student_id`
- `list` — вывод списка студентов с ограничением по количеству

Примеры использования:

```bash
python3 cli.py seed --count 1000
python3 cli.py add --student-id S999999 --full-name "Test Student" --group A-101 --faculty DE --admission-year 2026
python3 cli.py get --student-id S000000
python3 cli.py list --limit 3
```

Такой набор команд выбран как минимально достаточный для демонстрации работы приложения с шардированной БД.

## 6. Нагрузочное тестирование

Для нагрузочного тестирования реализован скрипт `benchmark.py`.

### Сценарий теста

Скрипт выполняет два основных типа операций:

- вставка `5000` новых документов в коллекцию `students`;
- выполнение `1000` точечных чтений по полю `student_id`.

Перед запуском теста в коллекцию были загружены тестовые данные через команду:

```bash
python3 cli.py seed --count 1000
```

Во время проверки проекта были получены следующие результаты:

### Визуализация результатов нагрузочного тестирования

Таблица результатов:

| Сценарий | Результат |
|---|---:|
| Вставка 5000 документов, прогон 1 | 29.94 c |
| Чтение 1000 документов, прогон 1 | 1.031 c |
| Вставка 5000 документов, прогон 2 | 33.184 c |
| Чтение 1000 документов, прогон 2 | 1.771 c |

## 7. Итог

По итогам нагрузочного тестирования можно сделать вывод, что выбранный хэшированный ключ шардинга обеспечивает почти равномерное распределение данных между двумя shard-ами и позволяет избежать перекоса хранения. Вставка большого числа документов занимает заметно больше времени, чем точечное чтение, что ожидаемо для такого сценария, однако чтения по `student_id` остаются быстрыми и стабильными. В целом результаты подтверждают, что даже в минимальной конфигурации шардинг работает корректно, данные действительно распределяются по узлам кластера, а система остаётся пригодной для базовых операций записи и чтения под нагрузкой.
